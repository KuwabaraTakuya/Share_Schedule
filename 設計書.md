## 1. プロジェクト概要

### 1.1 コンセプト

Slack の使いやすいチャンネル管理と、Google カレンダーのスケジュール機能を統合した、少人数グループ（数人〜10 人程度）向けのコミュニケーションツール。

グループメンバー全員の都合をカレンダー上で一目で確認（○/×/△）でき、日付ベースのチャンネル作成・場所共有・AI による予定入力など、スケジュール調整に特化した機能を持つ。

### 1.2 ターゲットユーザー

- 友人グループ（旅行・イベント計画）
- 小規模チーム・サークル・部活
- 家族グループ

### 1.3 主な特徴

- カレンダーファースト: トップ画面に全員の空き状況を表示
- マルチコミュニティ: 複数のグループに同時所属可能
- AI スケジュール入力: 音声・テキストで自然言語から予定を自動登録
- 場所共有: Google Maps から待ち合わせ場所をチャット内で共有
- 軽量・高速: Discord ライクな軽量動作
- セキュリティ重視: 個人のスケジュールを細かく公開設定可能

### 2.5 カレンダー機能

#### 2.5.1 コミュニティカレンダー（トップ画面）

| 機能 | 詳細 |
|------|------|
| 空き状況一覧 | 月ビューでメンバー全員の空き状況を ○/×/△ で表示 |
| ○/×/△ 自動判定 | LLM + ルールベースでメンバーの予定から自動算出（詳細は §8） |
| 日付クリック | クリック/タップでその日のメンバーの予定一覧と日付チャンネルを表示 |
| 日付チャンネル作成 | カレンダーの日付から直接、日付ベースのチャンネルを作成 |

#### 2.5.2 個人カレンダー

| 機能 | 詳細 |
|------|------|
| 月/週/日 ビュー | Google カレンダーライクな表示切り替え |
| 予定の手動入力 | タイトル・日時・場所・メモ・終日フラグ |
| 時間単位の予定 | 時間軸表示（週/日ビュー）で視覚的に入力 |
| 繰り返し予定 | 毎日・毎週・毎月・カスタム設定 |
| Google Calendar 連携 | 双方向同期（読み込み・書き込み） |
| iCal (.ics) 連携 | 外部カレンダーを読み込み専用で表示 |
| 音声入力 | マイクボタン押下で音声認識 → LLM で予定解析 → 確認画面 → 登録 |
| キーワード入力 | 自然言語テキストから予定自動生成（例: "2日 12時 学校"） |

#### 2.5.3 プライバシー設定

初期状態: 自分の予定は誰にも共有しない（自分だけが見える）

| 設定値 | 挙動 |
|--------|------|
| 全員に公開 | コミュニティ全メンバーが予定の詳細を確認可能 |
| 一部に公開 | 選択したメンバーのみが詳細を確認可能 |
| 共有しない（初期値） | ○/×/△ 判定の計算には使われるが、予定の詳細は他者に非表示 |

> **注意:** ○/×/△ の計算には全員の予定データを使用するが、表示名・予定内容は伏せて「何人が予定ありか」のみ他者に表示する。

## 3. 非機能要件

| 項目 | 要件 |
|------|------|
| パフォーマンス | メッセージ表示遅延 < 200ms（Firestore リアルタイム） |

| アクセシビリティ | WCAG 2.1 AA 準拠（キーボード操作・スクリーンリーダー対応） |

### 5.1 データフロー

**チャットメッセージ送信:**
```
ユーザー入力
  → Firestore (クライアント SDK から直接書き込み)
  → Firestore リアルタイムリスナーが他メンバーに即時配信
  → FCM でプッシュ通知（バックグラウンドのメンバー向け）
```

**カレンダー空き状況取得:**
```
画面ロード
  → Go API に空き状況計算をリクエスト
  → Go API がメンバー全員の予定データを Firestore から取得
  → Gemini API で ○/×/△ を判定
  → 結果をクライアントに返却 → カレンダーに表示
```

**予定の自然言語入力:**
```
ユーザーがテキスト or 音声入力
  → [音声の場合] Web Speech API でテキスト化
  → Go API にテキストを送信
  → Gemini API でスケジュールデータに変換
  → 確認画面をクライアントに返却
  → ユーザーが確認・修正 → Firestore に保存
  → (Google Calendar 連携中の場合) Calendar API にも書き込み
```

## 6. データベース設計

Firestore のコレクション構造。データはサブコレクションで階層化し、セキュリティルールで適切にアクセス制御する。

### 6.1 コレクション構造

```
users/
  {userId}
    displayName: string
    email: string
    avatarUrl: string
    timezone: string               // 例: "Asia/Tokyo"
    createdAt: timestamp
    updatedAt: timestamp
    settings:
      globalNotification:
        enabled: boolean
        quietHours: { start: string, end: string }  // "23:00", "07:00"
    calendarConnections: [         // Google Calendar 連携情報
      {
        provider: "google" | "ics"
        calendarId: string
        accessToken: string        // AES-256 で暗号化して保存
        refreshToken: string       // AES-256 で暗号化
        syncedAt: timestamp
      }
    ]

communities/
  {communityId}
    name: string
    iconUrl: string
    ownerId: string
    createdAt: timestamp
    inviteCode: string             // 招待リンク用コード
    inviteCodeExpiry: timestamp?
    settings:
      availabilityThreshold: number  // △ 判定の閾値(%) デフォルト: 80
      allowAnonymousAvailability: boolean  // 名前を伏せた△判定

    members/                       // サブコレクション
      {userId}
        role: "owner" | "admin" | "member"
        displayName: string        // コミュニティ内ニックネーム
        joinedAt: timestamp
        privacySettings:
          shareCalendarWith: "all" | "none" | string[]  // デフォルト: "none"
          shareDetailWith: "all" | "none" | string[]    // 詳細公開先
        notificationSettings:
          default: "all" | "mentions" | "muted"
        sharedCalendarIds: string[]  // このコミュニティに共有しているカレンダーID

channels/
  {channelId}
    communityId: string
    name: string
    type: "text" | "date"
    date: timestamp?               // 日付チャンネルの場合のみ
    position: number               // 表示順（ドラッグ&ドロップで変更）
    createdAt: timestamp
    createdBy: string
    topic: string?                 // チャンネルの説明
    pinnedLocations: [             // ピン留めされた場所
      { placeId: string, name: string, lat: number, lng: number }
    ]

    messages/                      // サブコレクション
      {messageId}
        userId: string
        content: string
        type: "text" | "image" | "file" | "location" | "system"
        attachments: [
          {
            name: string
            url: string            // Firebase Storage URL
            size: number
            mimeType: string
            isSafe: boolean        // Safe Browsing 検証結果
          }
        ]
        location: {                // 場所共有の場合
          placeId: string
          name: string
          address: string
          lat: number
          lng: number
          mapThumbnailUrl: string
        }?
        reactions: {               // { "👍": ["userId1", "userId2"] }
          [emoji: string]: string[]
        }
        threadCount: number
        editedAt: timestamp?
        deletedAt: timestamp?      // 論理削除
        createdAt: timestamp
        readBy: string[]           // 既読ユーザーID一覧

events/                            // カレンダーイベント
  {eventId}
    userId: string
    communityId: string?           // コミュニティ共有の場合
    title: string
    description: string
    startTime: timestamp
    endTime: timestamp
    isAllDay: boolean
    location: string?
    color: string?                 // 表示色
    recurrence: {                  // 繰り返し設定
      type: "none" | "daily" | "weekly" | "monthly" | "custom"
      until: timestamp?
      interval: number?
    }
    visibility: "private" | "community" | string[]  // 公開範囲
    source: "manual" | "google" | "ics"
    sourceEventId: string?         // 外部カレンダーの元 ID
    syncedAt: timestamp?
    createdAt: timestamp
    updatedAt: timestamp

availability_cache/                // ○/×/△ の計算結果キャッシュ
  {communityId_YYYYMM}            // コミュニティ × 年月 でキャッシュ
    days: {
      [date: string]: {            // "2026-07-01"
        status: "○" | "△" | "×"
        availableCount: number
        totalCount: number
        calculatedAt: timestamp
      }
    }
```

### 6.2 Firestore セキュリティルール概要

```javascript
// 基本方針:
// - users: 本人のみ読み書き可能
// - communities: メンバーのみ読み取り可能
// - channels/messages: そのコミュニティのメンバーのみ
// - events: 本人 + プライバシー設定に従った公開先
// - 管理者アクション: role が "owner" または "admin" のみ
// - キック等の破壊的操作: Go API 経由（Firebase Admin SDK で実行）
//   → クライアントからは直接書き込めない
```

## 7. セキュリティ設計

### 7.1 認証

- Firebase Auth で Google OAuth 2.0 / メール認証を実施
- ID トークン（JWT）を全 API リクエストの Authorization ヘッダーに付与
- Go API 側で Firebase Admin SDK を使ってトークンを毎回検証
- トークンの自動更新: Firebase SDK が処理（有効期限 1 時間）
- セッション管理: Firebase の永続化設定でブラウザセッション管理

### 7.2 Firestore セキュリティルール

- コミュニティデータへのアクセスはメンバーのみ
- メッセージ削除: 本人 or Admin のみ
- キック・チャンネル削除: Firestore には直接書き込めない（Go API 経由のみ）
- プライバシー設定が "none" のイベントは本人以外が取得できないようにルールで制御

### 7.3 ファイルセキュリティ

- Firebase Storage セキュリティルール: 認証済みユーザーのみアップロード可能
- ファイルサイズ制限: 画像 10MB / ファイル 50MB
- 実行可能ファイル（.exe / .sh / .bat / .apk 等）: フロントエンドでブロック + 警告表示
- URL 安全性: メッセージ内 URL を Go API 経由で Google Safe Browsing API に問い合わせ
- 結果をメッセージのメタデータに保存し、フロントで警告バナー表示

### 7.4 通信セキュリティ

- 全通信 HTTPS / TLS 1.3
- CORS: Cloud Run の Go API は許可オリジンのみ受け付け
- CSP ヘッダー: XSS 対策
- Rate Limiting: Cloud Run で IP ごとに API レート制限（Firebase の DoS 対策はクォータで管理）
- Helmet.js 相当のセキュリティヘッダー: Go の Fiber ミドルウェアで設定

### 7.5 カレンダーアクセストークンの保護

- Google Calendar の OAuth トークンは AES-256-GCM で暗号化して Firestore に保存
- 暗号化キーは Google Cloud Secret Manager で管理
- アクセストークンのリフレッシュは Go API 側でのみ実施（クライアントには返さない）

### 7.6 プライバシー保護

- ○/×/△ 判定: 計算はサーバーサイドのみ。他者に「空いているかどうか」だけを返し、予定の詳細は渡さない
- 名前を伏せた表示: △ の場合「X 人が調整が必要」と表示（誰かは非表示、コミュニティの設定で変更可）

## 8. LLM 設計

### 8.1 使用モデル

**Gemini 1.5 Flash** (Google AI Studio / Vertex AI)

### 8.2 スケジュール自然言語解析

**入力例:**
```
"2日の12時に学校"
"今週の土曜日午後3時から映画"
"来月1日終日 東京出張"
```

**Go API のプロンプト設計:**
```
システム:
あなたはスケジュール解析AIです。ユーザーの自然言語入力を構造化された
スケジュールデータ（JSON）に変換してください。
現在日時: {現在のISO8601時刻}
ユーザーのタイムゾーン: {timezone}

出力形式（JSON のみ返却）:
{
  "title": "予定のタイトル",
  "startTime": "ISO8601形式",
  "endTime": "ISO8601形式",
  "isAllDay": false,
  "location": "場所（あれば）",
  "description": "メモ（あれば）",
  "confidence": 0.0〜1.0  // 解析の確信度
}

不明な部分は null とし、confidence が 0.7 未満の場合は確認を促す。

ユーザー入力: "{input}"
```

**フロント確認フロー:**
1. ユーザーが入力
2. Go API → Gemini API で解析
3. 解析結果を確認画面で表示
4. ユーザーが修正 → 確定 → Firestore / Google Calendar に保存

### 8.3 音声入力フロー

```
マイクボタン押下
  → Web Speech API (SpeechRecognition) でリアルタイム音声テキスト化
  → テキストが確定したら §8.2 の解析フローへ

Web Speech API が使えない環境の場合:
  → 音声データを録音 → Go API に送信
  → Gemini API（音声対応）でテキスト化 → 解析フローへ
```

### 8.4 ○/×/△ 自動判定エンジン

**判定フロー:**

```
1. Go API がコミュニティメンバーのその日の予定一覧を取得
   （プライバシー設定に関係なく、サーバーサイドで全データを参照）

2. ルールベース事前判定:
   - 全員空き → ○（Gemini 不使用でも確定）
   - 全員予定あり → ×（確定）
   - その他 → Gemini に送信

3. Gemini 判定プロンプト:
   システム:
   グループの空き時間を分析し ○/△/× を判定してください。
   
   メンバー数: {N}人
   しきい値設定: {threshold}% 以上が空きなら △
   匿名モード: 有効（名前は "メンバーA" 等で匿名化）
   
   データ:
   {メンバーA}: 終日予定あり
   {メンバーB}: 12:00〜14:00 予定あり, 他は空き
   {メンバーC}: 空き
   ...
   
   出力（JSON）:
   {
     "status": "○" | "△" | "×",
     "availableCount": 数字,
     "totalCount": 数字,
     "bestTimeRanges": ["午前", "午後"] // 部分的に空きがある場合
   }

4. キャッシュ保存:
   - 結果を availability_cache に保存（TTL: 1時間 or 予定更新時に無効化）
```

**しきい値設定（コミュニティ設定で変更可能）:**

| 状況 | デフォルト判定 |
|------|-------------|
| 全員空き | ○ |
| 80%以上空き | △ |
| 80%未満空き | × |

## 9. カレンダー連携設計

### 9.1 Google Calendar 連携フロー

```
1. ユーザーが設定画面で「Google Calendar を連携」をクリック
2. Firebase Auth の Google OAuth に calendar スコープを追加して認可
3. アクセストークン・リフレッシュトークンを取得
4. AES-256-GCM で暗号化して Firestore に保存
5. Go API が Google Calendar API でイベントを取得
6. Firestore の events コレクションに保存（source: "google"）
7. 以降、定期的にバックグラウンドで同期（Cloud Scheduler + Cloud Run Job）
```

**同期の方向:**
- 読み込み: Google Calendar → ShareSchedule（常時）
- 書き込み: ShareSchedule → Google Calendar（ユーザーが選択した場合）

### 9.2 iCal (.ics) 連携

- 外部カレンダー（Apple Calendar 等）から .ics URL を登録
- Go API が定期的に URL を取得・解析して events に保存
- 読み込み専用（.ics への書き戻しはしない）

### 9.3 複数コミュニティでのカレンダー共有

- ユーザーは自分のカレンダー（Google / 手動作成）を複数コミュニティに共有可能
- コミュニティ設定の `sharedCalendarIds` で管理
- 同じカレンダーを複数コミュニティで共有しても、データは events コレクションで 1 本管理

## 補足: 使用するサービスの無料枠まとめ

| サービス | 無料枠 | 少人数グループでの見込み |
|---------|--------|----------------------|
| Firebase Firestore | 1GiB ストレージ / 50K 読み/日 | 十分 |
| Firebase Auth | 無料（メール認証含む） | 問題なし |
| Firebase Storage | 5GB / 1GB ダウンロード/日 | 十分 |
| FCM | 無料 | 問題なし |
| Google Cloud Run | 2M リクエスト/月 無料 | 十分 |
| Gemini 1.5 Flash | 15 RPM / 100万トークン/日 | 十分 |
| Google Calendar API | 1M クエリ/日 | 十分 |
| Google Maps JS API | $200/月 クレジット | 要監視 |
| Safe Browsing API | 10,000 リクエスト/日 | 十分 |

> Google Maps API のみ使用量によっては課金が発生する可能性がある。場所共有の頻度が低い少人数グループであれば $200/月の無料クレジット内に収まる見込み。
